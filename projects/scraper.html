What It Does

This pipeline collects, structures, and embeds every piece of public data from the TankPit gaming community — player profiles, leaderboard rankings, and 14 years of daily bulletin board posts — and transforms it into a searchable knowledge base that powers the TankPit Discord Bot's AI Q&A system.

The end product is bible_chunks.json — 16,914 text chunks, each paired with a 768-dimensional embedding vector from nomic-embed. When a user asks the bot a question, this file is what gets searched via cosine similarity.

crawl_profiles.py
Hits the TankPit API for all 13,250 registered players. Async with aiohttp, rate-limited, resume-capable via --resume flag. Outputs structured .txt profiles to scraped/profiles/.
leaderboard_crawler.py
Crawls 1,326 pages of the overall leaderboard plus 4 years of annual rankings (2017–2023). Writes to SQLite leaderboard_history table with year, rank, player ID, and stats.
bulletin_crawler.py
Playwright-based scraper for the community bulletin board — 5,160 days of posts from 2012 to present. Handles JS-rendered content, pagination, and duplicate detection.
rebuild_bible.py
Chunks all .txt files, sends each chunk to nomic-embed (port 8081) for vectorization, and writes bible_chunks.json with text + float32 embedding arrays. 0 embedding failures across 16,914 chunks.

Async Architecture

All three scrapers are built on Python's asyncio with bounded concurrency — a semaphore limits simultaneous requests so the target API isn't overloaded and the scraper doesn't get rate-limited mid-run.

Resume capability was critical — scraping 13,250 profiles takes hours, and a network blip or rate limit shouldn't restart from zero. Each scraper checks which IDs are already saved and skips them on rerun.

The Embedding Pipeline

Once all raw data is collected, rebuild_bible.py converts everything into vector embeddings using a local instance of nomic-embed-text-v1.5 running via llama.cpp on port 8081. No cloud API required.

Full Tech Stack

HTTP Scraping
aiohttp for async API calls. Connection pooling, retry logic with exponential backoff, and per-domain rate limiting via asyncio.Semaphore.
JS-Rendered Pages
Playwright (async) for pages that require JavaScript to render content. Headless Chromium, waiting for network idle before scraping DOM.
Database
SQLite (tankpit.db) — tanks table with 13,250 rows, leaderboard_history for year-by-year rankings, leaderboard_snapshots for point-in-time captures.
Embeddings
nomic-embed-text-v1.5 Q4_K_M via llama.cpp on port 8081. Outputs 768-dimensional float32 vectors. Full rebuild takes ~20–30 minutes locally.
Data Format
Structured .txt profiles for human readability and chunking flexibility. SQLite for structured queries. JSON for the final embedded knowledge base.
Environment
WSL2 / Ubuntu 24. Python venv in ~/tankpit-ai/. Runs on the same machine as the Discord bot and LLM servers — no cloud infrastructure needed.

What I Learned

Building a scraper that handles 13,250 profiles reliably is a different problem than building one that handles 100. At scale you need resume capability, structured error handling, progress tracking, and a clear understanding of what "done" means when some requests fail.

The bulletin board scraper was the hardest — 14 years of JS-rendered pages with inconsistent markup, missing dates, and session handling. Playwright added complexity but was necessary; aiohttp alone couldn't render the content.

The embedding pipeline taught me what "production RAG" actually means. Chunking strategy matters enormously — chunks too large lose precision, chunks too small lose context. The final 512-token chunks with 64-token overlap hit the right balance for this dataset.
